## Identified Bottleneck

Root cause: **test volume has grown 3× faster than split count was updated**.

The CI comment says "12,198 total tests / 20 splits = ~610 tests/split". Actual count as of 2026-04-21: **38,805 tests**. The split count was never updated.

With 38,805 tests / 20 splits = ~1,940 tests per split running on **2-worker GitHub ubuntu-latest runners** (2 CPUs), each split processes ~970 tests per worker — this produces the observed 14-18 min runtimes.

Confirmed via split 8 CI log:
```text
created: 2/2 workers
2 workers [1941 items]
```

Additional compounding factors:
1. **No `.test_durations` file** — pytest-split uses naive round-robin by count, not by duration. This creates unbalanced splits (split 8 has 1.3× the runtime of split 18).
2. **`--reruns=2`** in `addopts` — slow tests retry twice, multiplying runtime for flaky tests.
3. **`fetch-depth: 0`** on the test matrix checkout — full git history fetch on every split (no reason to need full history for tests).

---

## Proposed Fix

**Primary: Double the split count (20 → 40).**

This halves tests-per-split from ~1,940 to ~970, and halves worker load from ~970 to ~485 per worker. Expected per-split runtime: ~7-9 min (vs 14-18 min). Wall-clock CI time: ~9-10 min.

**Secondary: Save/restore `.test_durations` file via GitHub Actions cache.**

When pytest-split has timing data it balances splits by actual test duration rather than count. This eliminates outlier splits. The file is generated by adding `--store-durations` to the pytest run and caching the result between runs.

**Tertiary: Fix `fetch-depth: 0`.**

The test matrix job doesn't use git history. Changing to `fetch-depth: 1` saves 1-3s per split (minor, but free).

---

## Changes Implemented

1. Split count: `20 → 40` in strategy matrix and pytest `--splits` arg
2. `.test_durations` cache: save after first run, restore before subsequent runs via `actions/cache`
3. `fetch-depth: 0 → 1` on test-matrix checkout
4. Updated stale comment from "12,198 total tests" to "38,805 total tests"

**Expected outcome**: CI wall-clock time drops from ~24 min to ~9-10 min on merge_group.
